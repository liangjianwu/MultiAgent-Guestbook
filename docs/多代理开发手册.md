# 多代理开发手册

## 1. 目标

以结构化、可视化的小步循环模式运行多代理开发，确保每一步开发需求明确、代码可审查、缺陷可追溯，最终通过测试验收并部署。

## 2. 角色定义

| 角色 | 模型 | 职责 |
|---|---|---|
| 主控代理 | Python 脚本 + 轻量越权 | 调度各角色、超时控制、状态汇总、必要时介入干预 |
| PM | kimi-k3 | 分析需求并制定计划、检查开发结果、整理缺陷、决定是否进入下一迭代、每轮汇报各角色工作 |
| Coder | ornith | 每次只完成一个小开发任务，书写开发笔记 |
| Reviewer | kimi-k3 | 仅审查代码质量、是否符合需求、上次问题是否修复，**不修改代码** |
| Tester | ornith | 编写测试用例、执行测试、输出测试报告；测试用例代码问题可自行修复，运行时问题最多三轮修复；业务逻辑问题不修复，仅记录 |
| 部署角色 | ornith | 在 PM 确认开发和测试完成后，执行部署和验收测试 |

## 3. 阶段一：PM 需求分析与计划阶段

### 3.1 用户与主控代理的聊天式需求探讨

1. 用户告诉主控代理整体项目需求和目标
2. 主控代理与用户展开聊天式探讨
3. 主控代理将聊天内容整理为需求草稿
4. 需求草稿应包含：
   - 项目背景与目标
   - 功能列表
   - 非功能约束
   - 技术栈与运行环境
   - 不确定或需要确认的点

### 3.2 用户触发需求分析

当用户说「开启需求分析」时：

1. 主控代理将当前需求草稿交给 PM
2. PM 基于需求草稿进行：
   - 需求清理与分解
   - 识别主要功能和依赖关系
   - 识别风险和不确定点
   - 输出分析报告和初步计划
3. PM 将结果返还主控代理
4. 主控代理展示给用户看

### 3.3 需求迭代循环

1. 用户查看 PM 的分析和计划
2. 用户继续补充需求或提出疑问
3. 主控代理记录新增需求和变更，更新需求草稿
4. 主控代理记录变更历史，包括：
   - 变更内容
   - 变更原因
   - 变更时间
5. 用户再次说「开启需求分析」，则重复 3.2 和 3.3

### 3.4 用户说「开始开发」后

1. PM 将初步计划细化为小颗粒度的执行计划
2. 计划中必须根据进度适当插入测试安排
3. 计划格式：

```
coder: <file> - <一句话任务>
reviewer: <file> - <审查要点>
[...]
tester: <范围> - <测试内容>
[...]
```

4. PM 展示最终执行计划给主控和用户
5. 主控和用户确认无异议后，进入开发微循环

## 4. 阶段二：开发微循环

### 4.1 每次循环启动（PM 主导）

每个微循环开始时，PM 必须向主控代理和用户汇报：

1. 上次微循环结果
   - 完成了什么
   - 遗畜了什么问题
2. 本次微循环要做的事
   - 修复问题，还是开发新需求
   - 具体任务清单
3. 是否需要测试
   - 测试范围
   - 测试预期结果

主控代理或用户在任何时刻可以召停并介入修正需求或任务。

### 4.2 开发阶段（Coder 执行，PM 跟踪）

1. PM 将当次开发任务交给 Coder
2. Coder 每次只完成一个小任务（一个文件或一个小逻辑）
3. Coder 完成后书写开发笔记，返还给 PM
4. 开发笔记应包含：
   - 本次完成的内容
   - 修改或新增的文件清单
   - 是否进行了语法检查或快速验证
   - 遇到的问题和解决方案
5. PM 检查是否完成开发目标：
   - 如有遗漏，立即让 Coder 补充
   - 如已完成，进入审查阶段

### 4.3 审查阶段（Reviewer 执行，PM 跟踪）

1. PM 将已完成的代码交给 Reviewer
2. Reviewer 按以下清单审查：
   - **语法正确性**：代码是否能被 Node.js 正常解析
   - **符合需求**：当次导开发任务的目标是否达成
   - **风格一致性**：与已有代码风格保持统一
   - **边界情况**：空值、超长、非法输入等是否处理
   - **上次遗留问题**：之前提出的问题是否已修复
   - **可测试性**：代码是否支持后续测试
   - **前端/响应式检查**（如有 Web UI）：
     - 是否包含 `<meta name="viewport">`
     - 是否设置了防止横向滚动的 CSS（如 `overflow-x: hidden` / `max-width: 100%`）
     - 长内容是否会撑开容器（`word-break` / `overflow-wrap` / `min-width: 0`）
     - 移动端输入框字体是否过小（iOS 自动缩放问题）
     - 是否有针对小屏的 `@media` 调整
     - 是否避免 inline event handlers、使用事件委托
3. Reviewer 输出审查报告，必须包含：
   - 审查结论（通过/不通过）
   - 发现的所有问题
   - 每个问题的严重程度
4. **Reviewer 严禁修改代码**，只能提出问题和建议
5. PM 检查审查结果：
   - 问题很多：不融入新需求，开启修复迭代
   - 问题很少或没有：进入新需求开发迭代

### 4.4 测试安排

PM 在制定计划时应根据进度适当插入测试工作，不必等到整体开发结束。

测试触发时机：
- 一个功能模块完成后
- 多个相关修复迭代结束后
- 整体开发完成前

测试用例规范：
- 测试应覆盖正向流程和常见异常流程
- 测试应使用真实运行时条件，避引入过于复杂的 mock
- 如必须 mock，应确保 mock 与实际模型行为一致
- 测试应支持连续运行，测试之间不应互相依赖
- 测试结束后应清理测试数据

Tester 的修复边界：
- 测试用例代码问题：Tester 可自行修复
- 运行时问题（如缺少依赖、连接错误等）：Tester 最多尝试三轮修复
- 业务逻辑问题：Tester 不修复，仅记录问题并返回 PM

测试结果处理：
- 测试失败：返回 PM，插入修复迭代
- 测试通过：进入下一个开发迭代或结束开发

### 4.5 PM 汇报

每个微循环结束时，PM 必须汇报：

1. 上次微循环结果
2. Coder 完成的工作
3. Reviewer 发现的问题
4. Tester 的测试结果
5. 下一次微循环计划
6. 需要用户或主控注意的事项

## 5. 迭代类型

### 5.1 新需求开发迭代

- 上一迭代审查问题很少或没有
- PM 将新需求纳入下一迭代任务清单

### 5.2 修复迭代

- 审查问题较多
- 测试失败
- PM 不融入新需求，专注于修复已知问题

## 6. 主控代理干预规则

1. 当某个角色执行超时或达到最大迭代次数时，主控代理应介入
2. 主控代理介入后，可以：
   - 终止当前任务
   - 将问题汇总后交给 PM 重新规划
   - 在特殊情况下直接调整任务清单
3. **主控代理可以直接修改代码的情况**：
   - 子代理连续两次以上无法完成精确代码修改，或在修复中反复引入回归
   - 修改后，必须让 Reviewer 和/Tester 重新验证，并在开发笔记中说明主控代理介入的原因
4. 主控代理不应无差别地修改代码，只在子代理明显失效时补位

## 7. 用户参与规则

1. 用户在任何时刻可以召停当前循环
2. 用户可以要求修正需求或任务
3. 用户不需要审批 PM 每次产出的计划，但可以在执行前提出异议

## 7.1 前端开发与响应式特定规则

1. PM 在计划阶段必须确认产品类型：纯 API 还是带 Web UI。带 UI 的项目必须将首页、响应式适配纳入开发计划。
2. Coder 在完成前端页面后，开发笔记中应说明：
   - 页面结构和路由方式
   - 是否添加 viewport
   - 采用的响应式策略（比如流式布局、断点、抗溢出处理）
3. Reviewer 必须按前端/响应式检查清单审查。
4. Tester 验收 Web 页面时，应：
   - 访问 `GET /` 验证 200
   - 使用浏览器开发者工具（如 Chrome DevTools）模拟移动端视口
   - 检查页面不能左右滚动
   - 检查表单、列表、删除等交互


## 8. 输出物

每个角色的产出应保存在对应目录：

- 主控：`需求草稿.md`、`变更历史.md`、`执行状态.md`
- PM：`pm-output/plan.md`、`pm-output/prd.md`、`pm-output/final-report.md`
- Coder：代码文件 + 开发笔记（可写入 `dev-output/`）
- Reviewer：`review-output/step-{n}.md`
- Tester：`qa-output/qa-report.md`、`qa-output/test-output.log`

## 9. 时间管理

- 总时间限制由主控代理统一管理
- 每个角色单次调用设置超时，超时后主控代理介入
- 不超过用户给定的总时间上限

## 10. 轮转结束条件

开发微循环结束后，进入部署验收阶段：

1. PM 确认所有计划中的开发任务已完成
2. PM 确认所有审查问题已修复
3. PM 确认测试已通过
4. PM 触发部署角色执行部署和验收测试
5. 部署角色完成部署并运行验收测试
6. 如部署或验收中发现代码功能或逻辑问题，返回开发微循环进行修复
7. 一切正常后，PM 输出最终报告

## 11. 部署验收标准

部署角色完成部署后，必须进行以下验收测试：

1. 服务器进程已正常启动
2. 监听的端口正确
3. 健康检查接口返回 200 OK
4. 核心业务接口可正常调用
5. 数据库表已存在且可正常读写
6. 日志无未处理错误
7. **Web 页面验收**（如有 Web UI）：
   - `GET /` 能返回页面并返回 200
   - 页面正常渲染，无明显 404 / 500 / JS 报错
   - 使用 Chrome DevTools 或类似工具模拟手机视口，确认页面不能左右滚动
   - 检查 `<meta name="viewport">` 是否存在
   - 表单提交、列表加载、删除等交互可正常工作

验收失败时，部署角色应详细记录环境信息、部署步骤、错误输出和复现方法，返回 PM。

## 12. 执行中的额外规则

### 12.1 代码修复责任归属

- 审查问题：由 PM 安排 Coder 修复，Reviewer 不修改
- 测试用例问题：由 Tester 自行修复
- 运行时问题：由 Tester 尝试修复，超过三轮则返回 PM
- 业务逻辑问题：由 Tester 记录并返回 PM，PM 安排 Coder 修复
- 部署环境问题：由部署角色记录，如需要代码修改则返回 PM

### 12.2 缺陷跟踪

- 每个运行阶段完成后，PM 应更新当前开发状态和已知问题列表
- 已知问题应记录问题内容、责任角色、计划修复时间
- 问题修复后应进行验证，确认真正解决

### 12.3 最终验收

- 最终验收应在真实部署环境中进行
- 测试通过不等于上线成功，应直接测试运行中的服务
- 验收失败应详细记录环境信息、操作步骤和错误输出

## 13. 优化方向

每次项目结束后，应根据执行过程中的问题和经验，对以下方面进行优化：

- 步骤粒度
- 提示词质量
- 超时策略
- 角色能力边界
- 审查规则
- 测试触发时机
- 部署验收标准
- 缺陷跟踪机制
